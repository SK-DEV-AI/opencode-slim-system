# Architecture

## Overview

opencode-slim-system is an OpenCode plugin that reduces per-request token overhead by replacing the stock system prompt and built-in tool descriptions with compact versions. It hooks into two OpenCode plugin hooks, a TUI sidebar slot, and supports per-model customization.

## Plugin Hooks

### `tool.definition` (server plugin)

Fires once per tool when the LLM session starts. The callback receives each tool's `toolID` and an `output` object with a mutable `description` field. If we have a slim description for that tool ID (from the config dir, custom toolsDir, or embedded defaults), we replace `output.description` with it.

**Why a hook instead of a file override:** OpenCode's built-in tool descriptions are compiled into the binary as template strings stored in `packages/opencode/src/tool/*.txt`. These templates are rendered at runtime with platform/session context (OS, shell, limits, working directory). There is no filesystem override mechanism — the `~/.config/opencode/tool/` feature proposed in PR #7264 was not merged. The `tool.definition` hook is the only way to swap tool descriptions.

**Priority chain:** `options.tools[toolID]` (inline override) > `toolsDir/{toolID}.{modelKey}.txt` (per-model) > `toolsDir/{toolID}.txt` (custom dir) > `~/.config/opencode/slim-system/tool/{toolID}.txt` (config dir) > `DEFAULT_TOOL_DESCRIPTIONS` (embedded fallback) > stock description (no match).

### `experimental.chat.system.transform` (server plugin)

Fires when the system prompt is assembled for the LLM. Checks each prompt segment against a set of marker strings (e.g., "best coding agent on the planet") that identify stock OpenCode prompts. If a segment matches, it replaces the stock prompt with our slim version, preserving the environment block (model info, OS, directory, date — everything after `Instructions from:`).

**Marker heuristic:** The plugin looks for specific strings that appear in OpenCode's stock prompt files. Custom prompts (agent `.md` files, user-edited prompts) won't contain these markers and are left untouched.

**Model-specific prompts:** If a file `prompt/{modelKey}.txt` exists in the config directory, it takes precedence over `prompt/default.txt`.

## Key Modules

### `src/index.ts` — Server Plugin

Entry point exported as default from the package. Exports:

- `default` (async function): Main plugin factory. Accepts `PluginInput` and `PluginOptions`. Returns `Hooks` object with `tool.definition`, `experimental.chat.system.transform`, and `experimental.session.compacting` hooks. Reads config dir, resolves placeholders, builds status, triggers async npm version check.
- `parseModelFromFile(configPath)`: Parses JSONC file to extract the `"model"` field. Uses regex-based comment stripping (block `/* */` and line `//`) before matching the key. Returns the model string or `"unknown"`.

**Module-level state:** Unlike the previous version where `seedConfigDir()` ran at import time, all state initialization is now deferred to the `plugin()` factory function. This avoids side effects during import and makes the module testable without seeding the user's config directory.

**Status file:** On init, writes `/tmp/opencode-slim-system.json` with tool count, version info, and model key. The TUI sidebar polls this file.

**Version check:** Background fetch to npm registry at startup. Compares `dist-tags.latest` against installed version using the `semver` npm package (`coerce()` + `gt()` for robust handling of version strings).

### `src/constants.ts` — Shared Constants

Single source of truth for paths and values shared across `src/index.ts`, `tui/index.tsx`, and CLI tools.

- `STATUS_FILE`, `LOG_FILE`: Ephemeral runtime files in `/tmp/`
- `CACHE_DIR`, `ANNOUNCED_FILE`: XDG state directory for update notification persistence
- `CONFIG_DIR`, `CONFIG_TOOLS_DIR`, `CONFIG_PROMPT_DIR`, `CONFIG_PROMPT_FILE`: User-editable config location
- `PLACEHOLDERS`: Registry of template placeholders with resolver functions (year, OS, shell, limits)
- `BASE_TOOL_IDS`: 16 known tool IDs corresponding to current OpenCode built-in tools
- `DEFAULT_PROMPT_MARKERS`, `ENV_BLOCK_MARKERS`: Heuristic strings for stock prompt detection

### `src/defaults.ts` — Embedded Defaults

Auto-generated by `scripts/gen-defaults.js`. Contains:

- `DEFAULT_SYSTEM_PROMPT`: The slim system prompt (~240 tokens)
- `DEFAULT_TOOL_DESCRIPTIONS`: 16 tool descriptions (~3,650 chars total)

These are embedded in the published package and used as fallback when the config directory is empty or unavailable. The generator script reads from `tool/*.txt` and `prompt/default.txt` and produces a TypeScript file with properly escaped template literals.

### `tui/index.tsx` — TUI Sidebar Plugin

Registered at sidebar order 899. Renders a status panel showing:

- Tools slimmed count
- Estimated tokens saved per request (approximate)
- Update available indicator (⬆) when a newer npm version exists

Uses a 5-second polling interval to read `/tmp/opencode-slim-system.json`. Compares the expected npm version against the one embedded in the status file to detect stale data from previous sessions.

**Update dialog:** On first TUI start per version (checked against `~/.local/state/opencode-slim-system/announced.json`), shows a dialog with current vs latest version and a link to GitHub releases. Clicking confirm or dismissing silences it for that version.

### `scripts/` — Build and Maintenance Tools

| Script | Purpose |
|--------|---------|
| `gen-defaults.js` | Reads `tool/*.txt` and `prompt/default.txt`, writes `src/defaults.ts` with embedded defaults |
| `drift-check.js` | Verifies all known OpenCode tool IDs have corresponding `.txt` files in `tool/`. Used in CI. |

### `cli/` — CLI Tools

| Command | Function |
|---------|----------|
| `opencode-slim-export` | Export config directory as JSON blob (backup) |
| `opencode-slim-import <file>` | Import JSON blob into config directory (restore) |
| `opencode-slim-dump` | Dump embedded defaults as JSON (ignores user edits) |
| `opencode-slim-check` | Drift check alias for `scripts/drift-check.js` |

## Data Flow

```
OpenCode startup
  │
  ├── Server plugin loads (src/index.ts)
  │     ├── seedConfigDir() — creates tool/*.txt and prompt/default.txt if missing
  │     ├── readToolsFromDir() — loads all .txt files, resolves placeholders
  │     ├── getCurrentModel() — parses opencode.jsonc "model" field
  │     ├── buildStatus() — writes /tmp/opencode-slim-system.json
  │     └── checkLatestVersion() — async npm registry fetch
  │
  ├── tool.definition hook fires per tool
  │     └── resolveTool() — priority chain: inline > per-model > custom dir > config dir > defaults
  │
  ├── experimental.chat.system.transform hook
  │     └── detect stock prompt → replace with slim version, preserve env block
  │
  └── TUI loads (tui/index.tsx)
        └── poll /tmp/opencode-slim-system.json every 5s → update sidebar
```

## Testing

Tests are in `tests/` using Vitest. They cover:

- **Placeholder resolution:** All 8 template placeholders resolve to correct values
- **JSONC parsing:** `parseModelFromFile()` with clean JSON, JSONC with comments, missing field, nonexistent file
- **File I/O:** `readToolsFromDir()` and `writeDirFromMap()` with temp directories
- **Default integrity:** Every `tool/*.txt` file has a matching entry in `DEFAULT_TOOL_DESCRIPTIONS` with identical content
- **Tool ID completeness:** `BASE_TOOL_IDS` matches every key in `DEFAULT_TOOL_DESCRIPTIONS`
- **System prompt:** Contains key behavioral instructions, no raw placeholders
- **Tool description quality:** No unbalanced backticks, no unresolved placeholders, non-empty

Run with: `npx vitest run`
